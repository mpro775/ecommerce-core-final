# خطة تحويل نسخة النظام إلى متجر إلكتروني عام مستقل

## 1. الهدف العام

إنشاء منتج مستقل باسم مؤقت:

```text
General Ecommerce Core
```

يكون عبارة عن نواة متجر إلكتروني كاملة يمكن نسخها وتخصيصها لأي عميل جديد.

يتكون المنتج النهائي من:

```text
ecommerce-core-api
ecommerce-core-admin
ecommerce-core-mobile
```

ويشمل:

- Backend مستقل.
- لوحة تحكم مستقلة.
- تطبيق Flutter مستقل.
- PostgreSQL.
- Redis.
- Object Storage.
- Firebase Notifications.
- نظام إعدادات وهوية قابل للتخصيص.
- أدوات تهيئة متجر جديد.
- اختبارات شاملة لدورة التجارة الإلكترونية.

المنتج الجديد لا يكون SaaS، ولا يحتوي على:

- تسجيل أصحاب متاجر.
- إنشاء متاجر ذاتيًا.
- اشتراكات وباقات.
- إدارة منصة مركزية.
- قوالب ويب.
- منشئ صفحات.
- Storefront Web Builder.

---

## 2. النتيجة المطلوبة

بعد اكتمال الخطة يجب أن نستطيع إنشاء متجر جديد بهذه العملية:

```text
نسخ النواة
↓
إدخال بيانات العميل
↓
تحديد الهوية والعملات والميزات
↓
تشغيل قاعدة البيانات
↓
إنشاء حساب المالك
↓
تشغيل Backend ولوحة التحكم
↓
بناء تطبيق Android وiOS
```

دون الحاجة إلى تعديل معماري عميق في كل مرة.

## 3. المبادئ المعمارية المعتمدة

### 3.1 متجر واحد لكل Deployment

كل نسخة مخصصة لعميل واحد تحتوي على:

- قاعدة بيانات مستقلة.
- متجر واحد.
- مستخدمين وموظفين تابعين له.
- عملاء ومنتجات وطلبات خاصة به.

يتم الاحتفاظ داخليًا بـ:

```text
stores
store_users
store_id
```

لكن لا توجد واجهات أو عمليات تسمح بإنشاء متجر ثانٍ.

### 3.2 قاعدة بيانات مستقلة لكل عميل

لكل عميل:

- PostgreSQL مستقلة.
- Redis مستقل أو Namespace مستقل.
- Bucket تخزين مستقل.
- Firebase Project مستقل.
- مفاتيح دفع وشحن مستقلة.

هذا يوفر:

- عزل البيانات.
- سهولة النسخ الاحتياطي.
- سهولة الصيانة.
- تخصيص كل عميل.
- تقليل مخاطر اختلاط البيانات.

### 3.3 التخصيص عبر Configuration

أي اختلاف بين العملاء يجب أن يمر عبر:

- Store Settings.
- Feature Flags.
- Branding.
- Integrations.
- Environment Variables.

ولا يتم وضع شروط خاصة بعميل داخل النواة العامة.

## 5. المرحلة الثانية: حذف نظام SaaS


### 5.2 حذف عمليات SaaS

يتم حذف:

- إنشاء متجر عند التسجيل.
- تسجيل أصحاب المتاجر العام.
- إدارة الباقات.
- حدود الاستخدام.
- Plan Entitlements.
- اشتراكات المتاجر.
- الفوترة الدورية.
- تجديد الاشتراكات.
- إيقاف المتجر بسبب انتهاء الاشتراك.
- كوبونات الاشتراك.
- إثباتات دفع الاشتراك.
- إدارة جميع المتاجر مركزيًا.
- Platform Support.
- Platform Risk Management.
- Platform Compliance.
- Platform Automation.
- SaaS Analytics.
- QA Runner.

### 5.3 تنظيف AppModule

يتم إزالة:

- Modules.
- Controllers.
- Providers.
- Workers.
- Cron Jobs.
- Event Consumers.
- Routes.
- Swagger Tags.

المتعلقة بالمنصة والاشتراكات.

### 5.4 استبدال قيود الباقات

قد تعتمد وحدات عامة على خدمات مثل:

```text
assertFeatureEnabled
isFeatureEnabled
assertMetricCanGrow
recordUsageEvent
ensureDefaultSubscription
```

يتم استبدالها بخدمة:

```text
StoreCapabilitiesService
```

مسؤولياتها:

- هل الميزة مفعلة؟
- هل يسمح المتجر باستخدامها؟
- هل تظهر في لوحة التحكم؟
- هل تظهر في التطبيق؟

مثال:

```json
{
  "features": {
    "loyalty": true,
    "affiliates": false,
    "advancedOffers": true,
    "multiWarehouse": true,
    "reviews": true,
    "productQuestions": true,
    "abandonedCarts": true,
    "digitalProducts": false
  }
}
```

لا توجد:

- حدود باقة.
- عدد منتجات مسموح.
- مطالبة بالترقية.
- فوترة.

### 5.5 حذف جداول SaaS

إزالة جداول مثل:

```text
plans
plan_limits
plan_entitlements
store_subscriptions
subscription_invoices
subscription_payments
subscription_coupons
platform_admin_users
platform_support_cases
platform_incidents
platform_risk_violations
platform_compliance_tasks
platform_automation_rules
qa_runs
qa_results
```

### مخرجات المرحلة

- Backend مخصص لمتجر واحد.
- لا توجد اشتراكات أو منصة مركزية.
- جميع الميزات تعتمد على إعدادات المتجر.


```text

```

```text

GET /app/products/:slug
GET /app/policies

```
